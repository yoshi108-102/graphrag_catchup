## テキストユニットの選択
`_find_most_related_text_unit_from_entities` の中身を追う。
ざっくり言うと「エンティティが出現したチャンクを拾いつつ、そのエンティティの1-hop隣接ノードと共起しているチャンクを優先する」ロジック。

### ① 各エンティティが出現するチャンクIDを集める
```python
text_units = [
    split_string_by_multi_markers(dp["source_id"], [GRAPH_FIELD_SEP])
    for dp in node_datas
]
```
ノードの`source_id`は`chunk-a<SEP>chunk-b<SEP>...`のように複数チャンクIDを区切り文字で連結したもの。
これをバラしてエンティティごとの「登場チャンクリスト」を得る。

### ② 各エンティティの1-hop隣接ノードを取得
```python
edges = await asyncio.gather(
    *[knowledge_graph_inst.get_node_edges(dp["entity_name"]) for dp in node_datas]
)
all_one_hop_nodes = set()
for this_edges in edges:
    if not this_edges:
        continue
    all_one_hop_nodes.update([e[1] for e in this_edges])
```
エッジの相手側`e[1]`を集めて、対象エンティティに隣接しているノードの集合を作る。

### ③ 1-hopノードのチャンクID逆引きを作る
```python
all_one_hop_nodes_data = await asyncio.gather(
    *[knowledge_graph_inst.get_node(e) for e in all_one_hop_nodes]
)
all_one_hop_text_units_lookup = {
    k: set(split_string_by_multi_markers(v["source_id"], [GRAPH_FIELD_SEP]))
    for k, v in zip(all_one_hop_nodes, all_one_hop_nodes_data)
    if v is not None
}
```
隣接ノードごとに「そのノードが出現するチャンク集合」を用意。後の共起判定の逆引き辞書。

### ④ 候補チャンクにスコアを付ける
```python
all_text_units_lookup = {}
for index, (this_text_units, this_edges) in enumerate(zip(text_units, edges)):
    for c_id in this_text_units:
        if c_id in all_text_units_lookup:
            continue
        relation_counts = 0
        for e in this_edges:
            if (
                e[1] in all_one_hop_text_units_lookup
                and c_id in all_one_hop_text_units_lookup[e[1]]
            ):
                relation_counts += 1
        all_text_units_lookup[c_id] = {
            "data": await text_chunks_db.get_by_id(c_id),
            "order": index,
            "relation_counts": relation_counts,
        }
```
- `order` = そのチャンクを最初に拾ったエンティティの順位（クエリとの類似度順で並んでいるので、小さいほど偉い）
- `relation_counts` = そのチャンクの中に「対象エンティティ + その隣接ノード」が何ペア共起しているか

つまり「単にエンティティが出てるだけ」のチャンクより「関係が同じチャンク内で文章化されている」チャンクを評価するという発想。
関係性を言語化してあるチャンクの方が、LLMに推論させる材料としては良質、というのは直感的に納得できる。

#### relation_counts の深掘り

**何を数えているのか？**
対象エンティティを`A`、そのチャンクを`c_id`、`A`から生えているエッジの相手ノードを`{B1, B2, ...}`とすると、
```
relation_counts(A, c_id) = |{ Bi | Bi は c_id にも出現する }|
```
つまり「`A`のエッジ相手のうち、同じチャンク`c_id`内に共起している個数」。

**`get_node_edges`の返り値形式**
```python
# _storage/gdb_networkx.py
async def get_node_edges(self, source_node_id: str):
    if self._graph.has_node(source_node_id):
        return list(self._graph.edges(source_node_id))
    return None
```
networkxの`edges(node)`は`[(node, neighbor1), (node, neighbor2), ...]`を返すので、
`e[0]`は常に自分、`e[1]`は相手。だから`e[1] in all_one_hop_text_units_lookup`で「相手ノード名でチャンク集合を引く」ができる。

**「先着エンティティでスコア確定」の挙動**
```python
for index, (this_text_units, this_edges) in enumerate(zip(text_units, edges)):
    for c_id in this_text_units:
        if c_id in all_text_units_lookup:  # ← 既にスコア済みならスキップ
            continue
```
もし同じチャンク`c_id`がエンティティ`A(order=0)`と`B(order=1)`の両方に含まれていても、
`A`の回で`c_id`がスコアされたら`B`の回ではスキップされる。
つまり`relation_counts(c_id)`は**`c_id`を最初に拾ったエンティティ（=クエリに最も近いエンティティ）の隣接と共起している数**しか見ない。
`B`の隣接が`c_id`に出ていても加算されない。

- 利点: クエリ類似度トップのエンティティ文脈を優先するのは筋が通る。
- 欠点: チャンクの「全体的な関係密度」は測れていない（`A`視点の部分集計）。

**具体例**
- クエリが`[A(order=0), B(order=1)]`にヒット
- `A`のエッジ相手: `{X, Y}`、`B`のエッジ相手: `{Z}`
- チャンク`c2`には`A, B, X, Z`が登場
- 逆引き: `X ∈ chunks={c2, ...}`, `Y ∈ chunks={...}`, `Z ∈ chunks={c2, ...}`

処理:
- `A`のループ: `c2`で`X`が`c2`に居る → `+1`、`Y`は`c2`にいない → `+0`。**relation_counts(c2) = 1**で確定。
- `B`のループ: `c2`は既にスコア済みなのでスキップ。`Z`との共起はカウントされない。

もし`B`視点でも評価したら`c2`は`Z`との共起で`+1`されたはずで、合計2になりえた。
でも実装上はそうしない。「最初のエンティティ視点」での局所的な共起数に留まる。

**重複カウントについて**
`for e in this_edges`は「エッジ単位」で回すので、`A`から同じ相手`X`に複数のエッジがあれば（多重グラフなら）`X`の共起が複数回カウントされる。
ただしnano-graphragはnetworkxの無向単純グラフで実装されているので、同じ(`A`, `X`)ペアは1本しか存在しない。
→ 実質「相手ノードのユニーク数」に一致する。

**順位決めにおける役割**
最終ソートは`(order, -relation_counts)`。`order`が先なので、
- 違うエンティティ由来なら`order`（= クエリ類似度順）が効く
- 同じエンティティ由来のチャンク同士の並び替えで`relation_counts`が効く

つまり`relation_counts`は「**エンティティ内のチャンク優先順を、関係共起で決めるタイブレーカー**」という役回り。
グローバルな重要度を決める指標というより、局所的な並び替え用。

**直感的まとめ**
- 「`A`という固有名詞が出てるチャンク」はいくつもある
- そのうちLLMに渡すなら、`A`と一緒に`A`の隣人`X`や`Y`も同じ段落に居るチャンクの方が情報が詰まっている
- なので`A`に付随するチャンク集合をそのロジックで並べ替える

言い換えると、relation_countsは「そのチャンクがエンティティ`A`の"関係の証拠"としてどれだけ濃いか」を測る局所的な濃度指標。

### ⑤ ソートしてトークン上限で切る
```python
all_text_units = sorted(
    all_text_units, key=lambda x: (x["order"], -x["relation_counts"])
)
all_text_units = truncate_list_by_token_size(
    all_text_units,
    key=lambda x: x["data"]["content"],
    max_token_size=query_param.local_max_token_for_text_unit,
)
```
- 第一キー`order`（昇順）: クエリに近いエンティティ由来のチャンクが先。
- 第二キー`-relation_counts`（降順）: 同じエンティティ内では、関係が文章化されているチャンクを優先。

## 関係（エッジ）の選択
`_find_most_related_edges_from_entities`。エンティティに付随するエッジから関連の強いものを選ぶ。

```python
async def _find_most_related_edges_from_entities(
    node_datas: list[dict],
    query_param: QueryParam,
    knowledge_graph_inst: BaseGraphStorage,
):
    all_related_edges = await asyncio.gather(
        *[knowledge_graph_inst.get_node_edges(dp["entity_name"]) for dp in node_datas]
    )
    all_edges = set()
    for this_edges in all_related_edges:
        all_edges.update([tuple(sorted(e)) for e in this_edges])
    all_edges = list(all_edges)
    all_edges_pack = await asyncio.gather(
        *[knowledge_graph_inst.get_edge(e[0], e[1]) for e in all_edges]
    )
    all_edges_degree = await asyncio.gather(
        *[knowledge_graph_inst.edge_degree(e[0], e[1]) for e in all_edges]
    )
    all_edges_data = [
        {"src_tgt": k, "rank": d, **v}
        for k, v, d in zip(all_edges, all_edges_pack, all_edges_degree)
        if v is not None
    ]
    all_edges_data = sorted(
        all_edges_data, key=lambda x: (x["rank"], x["weight"]), reverse=True
    )
    all_edges_data = truncate_list_by_token_size(
        all_edges_data,
        key=lambda x: x["description"],
        max_token_size=query_param.local_max_token_for_local_context,
    )
    return all_edges_data
```
ポイントは以下。

### エッジの重複排除
```python
all_edges.update([tuple(sorted(e)) for e in this_edges])
```
`tuple(sorted(e))`しているので、`(A, B)`と`(B, A)`は同一エッジとして扱われる（無向扱い）。
複数のエンティティから同じエッジを拾っても1本にまとまる。

### rank = edge_degree
```python
all_edges_degree = await asyncio.gather(
    *[knowledge_graph_inst.edge_degree(e[0], e[1]) for e in all_edges]
)
```
`edge_degree(u, v) = degree(u) + degree(v)`。
つまり両端ノードの接続数の合計。グラフ的に「中心性の高いハブ同士を繋ぐエッジ」ほど高スコア。

### ソート
```python
all_edges_data = sorted(
    all_edges_data, key=lambda x: (x["rank"], x["weight"]), reverse=True
)
```
- 第一キー`rank`: 上述のedge_degree
- 第二キー`weight`: エッジ固有の重み（抽出時にLLMが付けたもの、複数回出現するエッジは累積されているはず）

rankを優先しているので、グラフ構造としての「重要そうなエッジ」が先に来る。同点なら重みで決める。

### トークン上限で切る
`local_max_token_for_local_context`まで`description`の合計トークン数で切る。
エッジのdescription（関係の説明文）がコンテキストに入る想定。

構成は以下の4ブロック：

| --- | --- | --- |

| Entities | ヒットしたエンティティ | id, entity, type, description, rank |
| Relationships | 選ばれたエッジ | id, source, target, description, weight, rank |

- **エンティティ（点）**: クエリに直接近いもの
- **リレーション（線）**: そのエンティティを含むエッジで、グラフ中心性の高いもの
- **テキスト（素材）**: エンティティ+隣接が共起している原文
- **コミュニティレポート（俯瞰）**: エンティティが属するクラスタの要約

点・線・原文・要約の4階層を1つのコンテキストに詰めてLLMに渡すのがローカル検索、と理解しておけば良さそう。

## グローバル検索
`global_query`。ローカル検索が「特定の固有名詞」を起点にするのに対し、グローバルは「文書全体で何が言えるか」を問う。
素材として使うのは**コミュニティレポートのみ**（エンティティもエッジも原文も直接は見ない）。
map-reduce構造で、「複数のアナリストに部分ごとの要点を出させてから統合する」という発想。

### 全体フロー

```
1. コミュニティスキーマ取得(level <= query_param.level でフィルタ)
       │
2. occurrence で降順ソート → 上位 global_max_consider_community 件に絞る
       │
3. レポート本体を取得 & rating >= global_min_community_rating でフィルタ
       │
4. (occurrence, rating) で再ソート
       │
5. ★ MAP ★  トークン予算ごとにグループ分割 → 各グループで LLM にキーポイント抽出
       │        (global_map_rag_points プロンプト)
       │
6. ★ REDUCE ★ 全ポイントを score 降順で連結 → LLM で最終回答生成
                 (global_reduce_rag_response プロンプト)
```

### ① コミュニティの事前絞り込み
```python
community_schema = await knowledge_graph_inst.community_schema()
community_schema = {
    k: v for k, v in community_schema.items() if v["level"] <= query_param.level
}
...
sorted_community_schemas = sorted(
    community_schema.items(),
    key=lambda x: x[1]["occurrence"],
    reverse=True,
)
sorted_community_schemas = sorted_community_schemas[
    : query_param.global_max_consider_community
]
```
- `level <= query_param.level` で階層を制限（ローカル検索と同じ）。
- `occurrence` = `len(chunk_ids) / max_num_ids`。コミュニティに属するチャンク数の全体に対する割合。大きいほど「広く文書全体に関わるコミュニティ」。
- この時点ではまだレポート本体は取ってない（スキーマのみ）。上位N件に絞ってからKVから引く。無駄なI/Oを避けるため。

### ② レポート取得と品質フィルタ
```python
community_datas = await community_reports.get_by_ids(
    [k[0] for k in sorted_community_schemas]
)
community_datas = [c for c in community_datas if c is not None]
community_datas = [
    c
    for c in community_datas
    if c["report_json"].get("rating", 0) >= query_param.global_min_community_rating
]
community_datas = sorted(
    community_datas,
    key=lambda x: (x["occurrence"], x["report_json"].get("rating", 0)),
    reverse=True,
)
```
- `rating`はインデックス時にLLMが付けたコミュニティの重要度スコア。`global_min_community_rating`未満は除外。
- 最終ソートは`(occurrence, rating)`降順。大きさ優先、同じなら質で決める。

### ③ MAP: コミュニティを束にしてLLMに要点抽出させる
`_map_global_communities`。

```python
community_groups = []
while len(communities_data):
    this_group = truncate_list_by_token_size(
        communities_data,
        key=lambda x: x["report_string"],
        max_token_size=query_param.global_max_token_for_community_report,
    )
    community_groups.append(this_group)
    communities_data = communities_data[len(this_group) :]
```
全部のコミュニティを1コンテキストに入れると溢れるので、トークン予算（`global_max_token_for_community_report`）で分割してグループを作る。
`truncate_list_by_token_size`は「先頭から足し込んでいって予算を超えない範囲で切る」関数なので、これを繰り返すことで全データが過不足なくグループ化される（最後のwhileで残り全部消費）。

```python
async def _process(community_truncated_datas: list[CommunitySchema]) -> dict:
    communities_section_list = [["id", "content", "rating", "importance"]]
    for i, c in enumerate(community_truncated_datas):
        communities_section_list.append(
            [
                i,
                c["report_string"],
                c["report_json"].get("rating", 0),
                c["occurrence"],
            ]
        )
    community_context = list_of_list_to_csv(communities_section_list)
    sys_prompt_temp = PROMPTS["global_map_rag_points"]
    sys_prompt = sys_prompt_temp.format(context_data=community_context)
    response = await use_model_func(
        query,
        system_prompt=sys_prompt,
        **query_param.global_special_community_map_llm_kwargs,
    )
    data = use_string_json_convert_func(response)
    return data.get("points", [])

responses = await asyncio.gather(*[_process(c) for c in community_groups])
```
グループごとに並列でLLMを叩く。プロンプト`global_map_rag_points`は要するに
「以下のコミュニティレポートを見て、ユーザーの質問に答えるキーポイントをJSON配列で出せ。各ポイントには description と 0-100 の importance score を付けろ」
というもの。`{"points": [{"description": "...", "score": 80}, ...]}`形式で返ってくる。

このグループ1つ1つが「アナリスト」の単位。各アナリストが担当範囲のレポートを見て要点をまとめる。

### ④ REDUCE: 全ポイントを集約して最終回答
```python
final_support_points = []
for i, mc in enumerate(map_communities_points):
    for point in mc:
        if "description" not in point:
            continue
        final_support_points.append(
            {
                "analyst": i,
                "answer": point["description"],
                "score": point.get("score", 1),
            }
        )
final_support_points = [p for p in final_support_points if p["score"] > 0]
```
全アナリストの出力をフラット化。`score=0`（LLMが「分からない」と判断したもの）は切り捨てる。`analyst`インデックスでどのグループ由来かは保持。

```python
final_support_points = sorted(
    final_support_points, key=lambda x: x["score"], reverse=True
)
final_support_points = truncate_list_by_token_size(
    final_support_points,
    key=lambda x: x["answer"],
    max_token_size=query_param.global_max_token_for_community_report,
)
points_context = []
for dp in final_support_points:
    points_context.append(
        f"""----Analyst {dp['analyst']}----
Importance Score: {dp['score']}
{dp['answer']}
"""
    )
points_context = "\n".join(points_context)
```
スコア降順でソート、トークン予算で切って、「----Analyst N----」形式に整形。

```python
sys_prompt_temp = PROMPTS["global_reduce_rag_response"]
response = await use_model_func(
    query,
    sys_prompt_temp.format(
        report_data=points_context, response_type=query_param.response_type
    ),
)
```
最後にreduceプロンプトで最終回答を作る。プロンプトは
「複数アナリストのレポートを統合して最終回答を作れ。レポートは重要度降順。根拠のない情報は入れるな」
というもの。

### ローカルとの比較

| 観点 | ローカル検索 | グローバル検索 |
| --- | --- | --- |
| 入り口 | クエリembedding → エンティティ検索 | コミュニティスキーマを直接使う |
| 主な素材 | エンティティ / エッジ / 原文 / レポート | レポートのみ |
| 粒度 | 点（固有の何か） | 面（文書全体の傾向） |
| LLM呼び出し | 1回（コンテキスト統合→回答） | N+1回（map N件 + reduce 1件） |
| 得意な質問 | 「Xは誰か？」「XとYの関係は？」 | 「全体の主要テーマは？」「登場人物の対立構造は？」 |
| コスト | 低（1コール） | 高（map-reduceで並列LLM） |

### 感想・疑問点
- mapの時点で各アナリストが「ユーザーの質問」を見ているのがポイント。ただの要約ではなく質問に対する要点抽出。これで無関係な情報が早期にフィルタされる。
- reduceに渡るのはアナリストの「要点リスト」であり、元のレポートは渡らない。つまり情報は2段階で圧縮されている。精度とコストのトレードオフ。
- `occurrence`でソートしているので、グローバル検索はどちらかというと「広いトピック」を拾いやすい。ニッチな重要事項はrating高でも`occurrence`が小さいと`global_max_consider_community`で落ちる可能性がある。この挙動は意図通りなのかちょっと気になる。
- ローカルでは`(count, rating)`で並べていたのに対し、グローバルでは`(occurrence, rating)`。文字通り「局所的な重要度 vs 全体カバレッジ」の対比になっていて綺麗。
